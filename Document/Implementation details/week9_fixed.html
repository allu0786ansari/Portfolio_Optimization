This week's work is what interviewers mean when they ask "how do you handle model drift in production?"

What metrics we export
From FastAPI via prometheus_client: request_count (counter), request_latency_ms (histogram), portfolio_return_daily (gauge), rolling_sharpe_30d (gauge). The rolling Sharpe is the drift signal — when it drops below 0.5, we trigger retraining.
Prometheus pull model
Prometheus is pull-based — it calls GET /metrics on your FastAPI every 15 seconds and stores the time series. You never push metrics. This is the opposite of logging and interviewers specifically ask about this distinction.
Grafana alert → retraining
Grafana monitors rolling_sharpe_30d. When it drops below 0.5 for 10 consecutive minutes, a webhook fires to alert_trigger.py. That server calls airflow dags trigger. Fully automated — zero human intervention.
Airflow DAG structure
5 tasks in sequence: fetch_new_data → engineer_features → train_models → evaluate_champion → notify_result. Each is a PythonOperator. If the challenger beats the champion by 5% Sharpe, it gets promoted in MLflow registry automatically.
Champion-challenger logic
evaluate_champion loads current champion val_sharpe from MLflow, trains challenger, compares. If challenger_sharpe > champion_sharpe × 1.05, the challenger gets the "champion" alias. FastAPI hot-reload picks it up within 5 minutes.
Why a 5% threshold?
Retraining is expensive and a worse model could be promoted. The threshold prevents model churn — we only replace when there is statistically meaningful improvement. This is standard champion-challenger strategy in production ML.
